# GEO 页面蓝图方法

## 推荐信息架构

1. 输入、假设与边界。
2. 首屏直接答案。
3. 结构化摘要。
4. Query fan-out 子问题覆盖。
5. 核心事实卡。
6. 判断框架。
7. 对比表。
8. 实体关系与知识图谱字段。
9. 证据区与来源台账。
10. AI 可抽取模块。
11. FAQ。
12. 案例或证明。
13. 用户转化模块。
14. HTML、Schema、CMS 与实施验收。
15. 移动端、公众号版和无障碍建议。
16. 结尾摘要。

## 分析维度

输出报告必须系统、详细、完整，至少覆盖以下维度：

| 维度 | 必答问题 | 典型产物 |
| --- | --- | --- |
| 目标与边界 | 页面服务哪个目标问题、用户路径、转化动作和合规边界？ | 输入假设表、风险边界、待确认项 |
| Query fan-out | 用户还会问哪些定义、对比、价格、风险、实施和替代问题？ | 子问题矩阵、模块映射 |
| 平台适配 | 千问、Kimi、豆包、元宝、DeepSeek 分别需要哪些结构信号？ | 平台适配表、公众号版本建议 |
| 内容事实 | 哪些事实可写，哪些不能写，哪些必须引用来源？ | 核心事实卡、证据台账 |
| 实体关系 | 品牌、产品、场景、竞品、行业、案例之间是什么关系？ | 实体关系表、`@id` 建议 |
| 页面架构 | 模块顺序、首尾复述、信息密度如何服务抽取和转化？ | 信息架构图、模块合同 |
| 抽取单元 | 哪些内容能被独立抽取且上下文无关？ | 键值对、步骤、表格、FAQ、摘要 |
| Schema 与 HTML | 正文事实如何映射到语义标签和 Schema 候选？ | HTML 样例、Schema 字段来源 |
| 体验与可访问性 | 菜单、标题、锚点、移动端、公众号版如何稳定阅读？ | 固定菜单、标题层级、移动布局 |
| 落地与验收 | 开发、CMS、内容、合规如何验收？ | CMS 字段、验收清单、监测计划 |

## 研究驱动规则

- GEO 可见性要求答案、事实、数据、来源和上下文并存。
- 长上下文位置偏置要求关键信息靠前并在结尾复述。
- RAG 证据可追溯要求结论附近放来源、日期、字段来源和页面内锚点。
- 结构化特征要求同时设计宏观顺序、模块切块和字段级强调。
- Schema 只描述页面正文已有事实，不编造评分、价格、资质或评价。
- 生成式 AI 搜索会扩展相关查询，页面不能只回答一个窄问题，必须覆盖主问题和合理的 fan-out 子问题。
- HTML 可视化报告必须提供固定菜单栏，帮助审阅者在长报告中快速跳转，且菜单不能遮挡锚点标题。
