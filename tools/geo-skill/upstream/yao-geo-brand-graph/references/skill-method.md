# 方法说明

`yao-geo-brand-graph` 的目标不是输出一张漂亮脑图，而是把品牌事实整理成可检索、可验证、可维护的实体关系层。

## 工作模型

| 层级 | 处理动作 | 输出 |
| --- | --- | --- |
| 来源层 | 记录来源类型、定位符、日期、事实主张和核验状态。 | 来源账本 |
| 提及层 | 抽取品牌名、产品名、人名、地点、案例、行业词和场景词。 | mention 列表 |
| 候选层 | 合并相似提及，保留别名、旧名、英文名和易混项。 | candidate 列表 |
| 规范层 | 选定 canonical entity，记录消歧理由和证据。 | 实体清单 |
| 关系层 | 抽取带方向、谓词、证据和可信等级的关系边。 | 关系清单 |
| 来源核验层 | 对来源 URL 做可达性、页面标题、核验时间和访问限制采样。 | 真实数据来源核验 |
| 审计层 | 用权威参考、来源覆盖、真实来源核验、实体覆盖、关系证据、Schema 一致性、内容资产和监测闭环检查完整性。 | 系统分析矩阵 |
| 应用层 | 生成 Mermaid、JSON-LD、RDF 式三元组、测试场景和页面补强建议。 | 四件套报告 |

## 关系约束

- 关系必须有方向，例如 `品牌 -> 拥有 -> 产品`。
- 关系必须有证据 ID；没有证据的关系只能列为待确认或补强建议。
- 不把“领先”“高效”“创新”等营销形容词当作实体关系。
- 客户、合作伙伴、人物、授权数据和效果数字先过隐私检查，再进入图谱。

## 完整性扩展

报告必须先自检分析是否系统、详细、完整，再输出图谱。默认增加十个模块：

1. 权威参考与适用边界：说明 RDF、JSON-LD、PROV-O、SHACL、SKOS、Schema.org、Google 结构化数据规范和实体链接研究如何约束本报告。
2. 系统分析维度矩阵：覆盖品牌定义、实体消歧、产品体系、人物地点、客户案例、证据、Schema、国内 AI 平台监测和内容补强。
3. 来源覆盖矩阵：按官网、产品页、投资者关系、SEC/监管文件、官方案例、第三方资料区分覆盖对象和缺口。
4. 真实数据来源核验：记录来源 URL 的 HTTP 状态、页面标题、最后核验时间、真实数据判断和处理建议。
5. 实体覆盖矩阵：检查品牌、产品、服务、功能、技术、行业、用户、场景、客户、案例、证据、地点、时间是否完整。
6. 关系审计矩阵：检查拥有、提供、适用、合作、获得、引用、服务、对比、替代、来源、时间等关系是否有方向和证据。
7. Schema 与页面正文一致性：JSON-LD 只写页面可见、可核验事实，不能用结构化数据承载页面没写的营销主张。
8. 内容资产对齐计划：把图谱关系反推到官网、FAQ、百科页、公众号和知识库更新要求。
9. 国内 AI 平台监测闭环：为 DeepSeek、豆包、千问、Kimi、腾讯元宝等平台设计采样问题、指标、频率和纠偏动作。
10. 分析完整性自检清单：输出前记录覆盖、证据、隐私、消歧、排版和四件套存在性检查。
