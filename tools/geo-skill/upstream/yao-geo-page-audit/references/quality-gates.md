# 页面 GEO 诊断质量门

## 输入质量门

- 至少有目标 URL 或官网；没有 URL 时必须先让用户补充。
- 必须记录品牌名、目标关键词或目标问题；缺失时在报告中标为输入缺口。
- 输入首页时必须说明代表性一级页和二级页的选择依据。

## 诊断质量门

- 必须同时覆盖可抓取性、渲染、结构规范性、内容信号、AI 可抽取性、schema 一致性、移动/性能、证据质量八类问题。
- 必须包含权威证据台账，并标明观察、官方、标准、研究、推断或缺口。
- 必须包含公开答案素材问题集，说明每个高意图问题需要什么官方答案材料。
- 没有用户提供的 AI 平台采样时，必须省略平台召回、排名、答案频次和引用份额分析。
- 必须区分用户可见内容、爬虫可读内容和 AI 可抽取内容。
- 不把 schema 当万能方案；schema 必须与正文一致。
- 没有日志时不能判断真实抓取频次。
- 重要结论必须给出可信度，且不能把第三方文章当作官方功能或价格依据。

## 修复清单质量门

- 每个 P0/P1 建议必须有问题、证据、影响、动作、负责人、成本和验收方式。
- 代码层建议必须给示例字段、HTML 片段、JSON-LD 片段、meta 配置或复测命令中的至少一种。
- 内容建议必须给模块级结构，而不是只写“补充优质内容”。
- 优先级不能全部是 P0。

## 四件套质量门

- Word、PDF、HTML、Markdown 必须来自同一个 Markdown 内容源。
- HTML 必须符合 Kami 风格：暖纸面、油墨蓝强调、暖灰边框、紧凑行距、可打印、移动端可读，表格不得溢出页面，并包含固定跟随的目录菜单栏。
- PDF 不得裁切宽表和代码块。
- Word 表格必须可编辑且不得向右溢出。
- Markdown 必须保留关键表格、代码块和证据台账。
- 所有 README 或 docs 中出现的示例链接必须有真实文件。
- 示例目录必须附带 `quality-report.json`，其中 `artifacts` 和 `layout_checks` 字段逐项记录四件套状态和排版检查。
- `quality-report.json` 必须记录 HTML sticky TOC 检查结果。
- `quality-report.json` 必须记录 Kami 风格基础检查结果，包括暖纸面、无 rgba、油墨蓝强调和正文行距。
